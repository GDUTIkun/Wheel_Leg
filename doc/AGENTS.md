# AGENTS.md

本文件是本项目给 Codex / Vibe Coding Agent 的长期工作规则。

本项目采用迭代式机器人开发流程。
不要一次性完成所有功能设计和实现。
每次只围绕一个明确、可验证的小目标进行开发。

机器人功能必须经过仿真或实机验证。
代码写完不等于功能完成。
未经验证的功能只能标记为“待验证”，不能标记为“已完成”。

---

## 1. 总体工作原则

1. 不要猜测用户意图。
2. 任何不明确的需求、接口、硬件约束、控制逻辑、验证标准，都必须先向用户提问。
3. 不要为了文档完整性虚构功能、接口、参数、通信协议或硬件连接方式。
4. 不要一次性实现未来功能。
5. 每次只处理当前迭代中的一个或少量最小任务。
6. 每次修改前先说明计划。
7. 每次修改后说明修改内容、影响范围和验证方法。
8. 如果发现任务过大，必须先拆分任务，而不是直接实现。
9. 如果测试失败，应先记录现象和分析原因，不要直接大范围重构。
10. 重要设计选择必须记录到 `doc/decisions.md`。

---

## 2. 推荐文档结构

项目文档放在 `doc/` 目录下：

```text
doc/
  proposal.md
  architecture.md
  detail.md
  decisions.md
  validation.md
  prompt.md
  iterations/
    iter-001.md
    iter-002.md
  tasks/
    progress.md
    <module-name>.md
```

各文件职责如下：

```text
proposal.md      管“项目要做什么”
architecture.md  管“系统怎么分层”
detail.md        管“当前迭代怎么实现”
iterations/      管“每一轮迭代目标、范围、验证标准”
tasks/           管“当前迭代的最小任务”
validation.md    管“仿真和实机跑出来怎么样”
decisions.md     管“为什么当时这样选”
prompt.md        管“每次给 Vibe Coding 的启动 Prompt”
```

---

## 3. 文档生成顺序

当用户要求生成开发文档时，按以下顺序进行：

1. `doc/proposal.md`
2. `doc/architecture.md`
3. `doc/iterations/iter-XXX.md`
4. `doc/detail.md`
5. `doc/tasks/<module-name>.md`
6. `doc/tasks/progress.md`
7. `doc/validation.md`
8. `doc/decisions.md`
9. `doc/prompt.md`

如果前置文档不存在，不要直接跳过。
应先提醒用户缺少哪些文档，并询问是否需要先生成。

---

## 4. 需求文档规则：doc/proposal.md

`doc/proposal.md` 用于记录项目总需求。

必须包含：

1. 项目背景
2. 总体目标
3. 当前已知硬件平台
4. 当前已知软件平台
5. 功能需求
6. 非功能需求
7. 仿真验证需求
8. 实机验证需求
9. 当前不确定问题
10. 暂不实现的内容

生成或更新需求文档时，必须确认：

1. 机器人类型
2. 主控平台
3. 传感器
4. 电机和驱动器
5. 通信方式
6. ROS2、STM32、树莓派、MuJoCo 的分工
7. 当前优先实现的功能
8. 每个功能的完成标准
9. 是否需要仿真和实机共用接口
10. 哪些内容已经确定，哪些内容只是设想

不明确的地方必须向用户提问。

---

## 5. 总体架构规则：doc/architecture.md

`doc/architecture.md` 用于记录长期架构，不要求一次性最终确定。

必须包含：

1. 系统总体架构
2. 硬件分层
3. 软件分层
4. ROS2 节点划分
5. STM32 与树莓派任务划分
6. 仿真环境与实机环境的接口对应关系
7. 数据流
8. 控制流
9. 模块边界
10. 当前不确定设计
11. 后续可能调整的部分

如果以下内容不明确，必须向用户提问：

1. 模块边界
2. 通信接口
3. 控制频率
4. ROS2 topic 名称
5. ROS2 message 类型
6. 串口协议
7. 遥控器接入方式
8. 传感器读取方式
9. 电机控制方式
10. 仿真和实机如何切换

不要虚构接口。

---

## 6. 当前迭代规则：doc/iterations/iter-XXX.md

每次开发必须围绕一个最小可验证迭代。

每个迭代文档必须包含：

1. 本次迭代目标
2. 本次不做什么
3. 涉及模块
4. 输入条件
5. 输出结果
6. 最小实现任务
7. 仿真验证方法
8. 实机验证方法
9. 通过标准
10. 风险点
11. 需要用户确认的问题

当前迭代只能包含当前可验证的小目标。

如果用户给出的目标过大，必须主动拆小。

示例：

不推荐：

```text
完成整套轮腿机器人运动控制系统。
```

推荐：

```text
完成遥控器通道值读取，并在 ROS2 topic 中稳定发布。
```

不推荐：

```text
完成 MuJoCo 到实机的完整迁移。
```

推荐：

```text
让仿真环境和实机环境共用同一套电机输出接口抽象。
```

## 7. 详细设计规则：doc/detail.md

`doc/detail.md` 根据需求文档、总体架构和当前迭代目标生成。

只细化当前迭代涉及的模块。
未进入当前迭代的模块只保留概览，不展开实现细节。

每个模块必须包含：

1. 模块目标
2. 模块输入
3. 模块输出
4. 模块内部逻辑
5. 与其他模块的接口
6. 仿真环境下的行为
7. 实机环境下的行为
8. 参数配置
9. 错误处理
10. 测试与验证方法
11. 当前不确定问题

如果接口、参数、频率、协议、控制逻辑或硬件连接方式不明确，必须先问用户。

---

## 8. 任务划分规则：doc/tasks/

任务必须足够小，可以被一次 Vibe Coding 实现或验证。

每个模块对应一个任务文件：

```text
doc/tasks/<module-name>.md
```

总体进度记录在：

```text
doc/tasks/progress.md
```

每个模块任务文件必须包含：

1. 模块名称
2. 当前迭代目标
3. 任务 checklist
4. 每个任务的输入
5. 每个任务的输出
6. 完成标准
7. 验证方法
8. 当前状态

任务状态使用：

```text
[ ] 未开始
[~] 进行中
[x] 代码已完成
[!] 阻塞或待验证
[?] 待确认
[v] 已通过验证
```

注意：

```text
[x] 代码已完成
```

不等于：

```text
[v] 已通过验证
```

只有通过仿真验证或实机验证后，才能标记为 `[v] 已通过验证`。

## 9. 验证记录规则：doc/validation.md

`doc/validation.md` 记录仿真和实机测试结果。

每条验证记录必须包含：

1. 日期
2. 对应迭代
3. 测试环境：仿真 / 实机
4. 测试目标
5. 测试步骤
6. 观察结果
7. 是否通过
8. 问题现象
9. 可能原因
10. 下一步建议

不要根据代码逻辑假设测试通过。
必须根据用户提供的实际测试结果更新验证状态。

如果用户给出日志、截图、视频现象或终端输出，应先记录到 `validation.md`，再分析是否需要修改设计或代码。

## 10. 技术决策规则：doc/decisions.md

`doc/decisions.md` 记录重要设计选择。

每条记录格式：

```text
## DEC-XXX: 决策标题

- 日期：
- 背景：
- 可选方案：
- 最终选择：
- 选择理由：
- 影响范围：
- 后续是否可修改：
```

适合记录的内容包括：

1. 遥控器接 STM32 还是树莓派
2. ROS2 topic 和 message 设计
3. MuJoCo 与实机接口如何统一
4. 控制器运行在树莓派还是 STM32
5. 传感器数据是否在 STM32 预处理
6. 电机控制协议
7. 坐标系约定
8. 控制频率选择
9. 仿真和实机如何切换
10. 安全保护策略

## 11. Vibe Coding Prompt 规则：doc/prompt.md

`doc/prompt.md` 是每次启动 Codex / Vibe Coding 的提示词。

它必须告诉 Agent：

1. 当前只实现一个迭代目标
2. 先阅读哪些文档
3. 不要全仓库盲目修改
4. 不要一次性实现未来功能
5. 不明确的问题必须先问
6. 修改前先给计划
7. 修改后更新任务状态
8. 修改后说明如何验证
9. 不能把未验证功能标记为完成

推荐内容：

```text
你是本项目的 Vibe Coding 编程助手。

请先阅读以下文件：

1. doc/proposal.md
2. doc/architecture.md
3. doc/detail.md
4. doc/iterations/ 中最新的迭代文档
5. doc/tasks/progress.md
6. 当前迭代涉及模块的任务文件
7. doc/validation.md
8. doc/decisions.md

工作规则：

1. 当前只处理最新迭代文档中的目标。
2. 不要实现未进入当前迭代的未来功能。
3. 不要全仓库盲目修改，只读取和当前任务相关的文件。
4. 修改前先说明计划，包括要读哪些文件、要改哪些文件、预计影响哪些模块。
5. 如果需求、接口、参数、硬件约束、验证标准不明确，必须先向用户提问。
6. 不要猜测用户意图。
7. 每次修改应尽量小，保证可以单独编译、运行或验证。
8. 修改完成后，必须输出：
   - 修改了哪些文件
   - 每个文件修改了什么
   - 为什么这样改
   - 如何运行
   - 如何仿真验证
   - 如何实机验证
   - 当前任务状态
9. 如果代码完成但尚未验证，只能标记为“待验证”，不能标记为“完成”。
10. 如果测试失败，请根据现象更新 doc/validation.md，不要直接大改设计。
11. 如果本次修改形成了重要技术选择，请更新 doc/decisions.md。
12. 如果发现任务过大，请先拆分任务，不要直接实现。

本次任务：

请读取最新迭代文档，选择其中一个最小可执行任务进行实现。
如果无法确定应该做哪个任务，请先列出可选任务，并向用户提问。
```

## 12. 代码修改规则

修改代码时必须遵守：

1. 先读相关文档，再读相关代码。
2. 不要全仓库盲目搜索和修改。
3. 不要一次性重构大量文件。
4. 不要修改与当前任务无关的模块。
5. 不要删除用户已有代码，除非明确说明原因并得到确认。
6. 新增接口时必须说明输入、输出和调用方。
7. 修改通信协议时必须同步更新文档。
8. 修改 ROS2 topic、message、参数名时必须同步更新文档。
9. 修改硬件相关逻辑时必须说明实机风险。
10. 涉及电机、遥控器、姿态控制、速度控制、安全保护的修改，必须给出安全测试步骤。

## 13. 机器人项目特殊规则

本项目可能涉及：

1. STM32
2. 树莓派
3. ROS2
4. MuJoCo
5. 航模遥控器和接收机
6. 电机控制
7. 传感器读取
8. 运动控制
9. 仿真到实机迁移

开发时必须优先考虑：

1. 仿真和实机接口一致
2. 上层控制逻辑不直接依赖具体硬件
3. 遥控输入、传感器输入、电机输出需要抽象接口
4. 实机测试必须有安全保护
5. 控制频率必须明确
6. 坐标系必须明确
7. 单位必须明确
8. 参数必须可配置
9. 每个模块必须可以单独验证

---

## 14. 仿真与实机接口原则

仿真环境和实机环境应尽量共用上层控制逻辑。

推荐抽象：

```text
遥控输入接口
传感器状态接口
机器人状态估计接口
控制器接口
电机命令输出接口
安全保护接口
```

MuJoCo 和实机的区别应尽量隔离在底层适配层。

不推荐：

```text
控制器直接读取 MuJoCo 内部变量
控制器直接写串口电机命令
仿真一套控制逻辑，实机另一套控制逻辑
```

推荐：

```text
控制器读取统一 RobotState
控制器输出统一 MotorCommand
仿真适配器负责 MuJoCo 数据转换
实机适配器负责传感器和电机通信
```

## 15. 子 Agent 使用规则

可以使用子 Agent 拆分任务，但子 Agent 不能用来无限扩大上下文。

子 Agent 适合处理：

1. 阅读某个模块
2. 分析某个 bug
3. 编写某个小功能
4. 补充某个模块测试
5. 检查某个接口设计

子 Agent 不适合：

1. 一次性理解整个项目
2. 同时修改多个强耦合模块
3. 替代主 Agent 的整体判断
4. 把多个上下文窗口简单叠加成更大的上下文

使用子 Agent 时，必须要求它输出摘要：

```text
1. 阅读了哪些文件
2. 修改了哪些文件
3. 关键结论
4. 风险点
5. 测试结果
6. 是否需要主 Agent 继续处理
```

主 Agent 只汇总必要摘要，不要把子 Agent 的全部上下文复制回来。

每次完成任务后，必须输出：

```text
## 本次完成内容

## 验证方法

- 编译命令：

- 实机验证：

## 当前状态

- 代码状态：
- 验证状态：
- 阻塞问题：

## 文档更新

- proposal.md：
- architecture.md：
- detail.md：
- tasks：
- validation.md：
- decisions.md：

## 下一步建议
```

```text
验证状态：待用户仿真 / 实机验证
```

不能写“已完成”。

## 17. 禁止行为

禁止以下行为：

1. 不提问就补全不明确需求。
2. 一次性生成过大的完整系统设计。
3. 一次性修改大量无关文件。
4. 把代码写完但未验证的任务标记为完成。
5. 忽略仿真和实机差异。
6. 跳过安全保护直接进行实机控制。
7. 虚构传感器、电机、协议、topic、message。
8. 为了看起来完整而写不存在的模块。
9. 未经确认改变硬件连接方案。
10. 未经确认改变控制架构。

---

## 18. 用户交互方式

当信息不明确时，用问题列表向用户确认。

提问应具体、可回答。

不推荐：

```text
请提供更多信息。
```

推荐：

```text
为了生成当前迭代文档，需要确认以下问题：

1. 本轮优先目标是遥控输入、MuJoCo 仿真接口，还是电机输出接口？
2. 遥控接收机当前计划接到 STM32 还是树莓派？
3. 当前是否已经有 ROS2 workspace？
4. 本轮完成标准是能打印通道值，还是要发布到 ROS2 topic？
5. 本轮是否需要实机验证？
```

---

## 19. 当前默认开发方式

除非用户另有说明，默认采用以下方式：

1. 先文档，后代码。
2. 先仿真，后实机。
3. 先接口抽象，后具体硬件适配。
4. 先最小闭环，后复杂控制。
5. 先单模块验证，后系统联调。
6. 先记录现象，后修改设计。
7. 先完成当前迭代，后进入下一迭代。
