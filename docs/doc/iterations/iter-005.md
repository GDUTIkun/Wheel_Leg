# Iteration 005: STM32 硬件接入与 100Hz 实机闭环准备

- 日期：2026-06-24
- 输入文档：`doc/tasks/500hz_to_100hz_balance_notes.md`、`doc/tasks/control_tuning.md`、`doc/proposal.md`
- 输出文档：`doc/tasks/stm32_hardware_integration.md`、`doc/tasks/progress.md`

## 1. 本次迭代目标

本次迭代目标是从 100Hz 仿真平衡基线转入 STM32 / 实机硬件接入阶段。

当前已确认：100Hz 控制频率下，MuJoCo 仿真可以在保留 `freejoint` 的前提下稳定站立。该结果先作为阶段性基线冻结，不再继续只围绕仿真做深度参数收敛。

下一阶段重点转为：

- 建立 Raspberry Pi 与 STM32 的通信闭环。
- 对齐真实 IMU、编码器、电机状态与 ROS2 状态接口。
- 对齐 ROS2 控制命令到 STM32 电机执行命令的映射。
- 验证 100Hz 实际控制周期、单位、方向、零位和安全保护。
- 在硬件闭环可观测、可停机、可回退的前提下重新小步调参。

## 2. 当前冻结输入

当前冻结输入包括：

- `500Hz -> 100Hz` 仿真平衡修改记录。
- 100Hz LQR + VMC + 腿长 PID 的仿真可站立基线。
- RC 遥控输入链路与 failsafe 行为。
- ROS2 侧 `/joint_command`、`/robot_state`、`/imu`、`/joint_states` 等接口边界。
- 当前控制参数只作为硬件接入初始参考，不视为实机最终参数。

## 3. 本次迭代范围

本次迭代包含：

- 明确 STM32 接入阶段的任务边界。
- 固定上硬件前必须检查的单位、方向、零位、限幅和周期。
- 固定实机初期调参原则：先小命令、先方向校验、先安全停机，再站立。
- 记录 100Hz 仿真阶段已暂停深调，等待硬件接入后再调参数。

## 4. 本次不做什么

本次迭代不做以下内容：

- 不承诺当前 100Hz 仿真参数就是实机最终参数。
- 不在硬件状态未校验前直接尝试完整站立。
- 不在通信和安全链路未闭环前扩大速度、转向或高度范围。
- 不把真实硬件差异继续用 MuJoCo 参数硬凑。

## 5. 推荐执行顺序

1. 先验证 Raspberry Pi 与 STM32 通信链路稳定。
2. 验证 STM32 上传的编码器、IMU、电机状态能映射到 ROS2 状态接口。
3. 验证 `/joint_command` 到 STM32 电机命令的方向、单位和限幅。
4. 悬空验证各关节和轮子的正方向。
5. 轻触地验证腿长支撑力方向和力矩限幅。
6. 确认 100Hz 控制周期稳定，记录周期抖动。
7. 再尝试低风险站立闭环。
8. 根据实测日志重新小步调整滤波、LQR 缩放和执行器输出限制。

## 6. 通过标准

阶段性通过标准：

- STM32 状态上报稳定，时间戳和周期可观测。
- IMU pitch / pitch_rate、编码器角度 / 角速度、电机反馈方向与仿真定义一致。
- ROS2 命令下发到 STM32 后，关节和轮子动作方向正确且受限幅保护。
- 通信异常、遥控 failsafe 或控制异常时可以进入安全状态。
- 在小命令、低风险条件下可以开始实机闭环调参。

## 7. 风险点

- 真实 IMU 零偏、安装方向和滤波延迟可能与仿真差异很大。
- 编码器速度估计噪声可能比仿真更明显，`phi_rate` 需要重新确认。
- 电机电流环、驱动限流和通信延迟会改变仿真中有效的力矩响应。
- 真实地面接触和轮胎摩擦会改变速度反馈、pitch 反馈和腿长支撑表现。
- 任一方向或单位错误都可能把 LQR 阻尼变成正反馈，必须先悬空和低力矩验证。

## 8. 当前记录结论

100Hz 仿真阶段先放置。后续参数调优以硬件接入后的实测现象为准，优先保证状态可信、命令可控、安全可退，再逐步恢复站立、速度、转向和高度能力。
