# 轮腿机器人系统需求方案

## 1. 项目背景与目标

本项目目标是开发一套轮腿机器人系统，软件架构需要同时支持 MuJoCo 仿真与后续真机部署。系统硬件规划如下：

- STM32：实时硬件层，负责电机驱动、编码器读取、IMU 读取、电流采样与底层安全保护。
- Raspberry Pi：ROS2 主控，负责状态估计、控制算法、模式管理与 ROS2 通信。
- 航模遥控器与接收机：作为人工控制输入源。
- 电机驱动与传感器：通过 STM32 或仿真硬件接口统一接入上层控制链路。

软件规划如下：

- ROS2 作为统一通信层。
- MuJoCo 作为主要仿真环境。
- 后续支持真机部署。
- 控制算法支持持续迭代，包括 LQR、MPC、WBC、状态机控制、强化学习策略与轨迹跟踪控制等。

核心设计原则是：同一套 ROS2 控制架构同时支持 MuJoCo 仿真和真机运行。控制器不直接依赖 MuJoCo API，也不直接访问 STM32 串口或 CAN。仿真与真机切换时，只替换硬件接口层，不修改控制算法层。

目标运行环境：

- ROS2 Jazzy。
- Ubuntu 24.04。
- Raspberry Pi 作为主控部署目标。
- 桌面 Linux 作为仿真开发目标。

## 2. 当前仓库现状

当前仓库中的 `transplant/` 目录作为迁移与参考材料来源，不应在本需求阶段主动修改。

### 2.1 `matlab_sim`

`matlab_sim` 用于前期控制算法验证、LQR 参数设计与数学模型验证。该目录已有一定算法积累，后续可以作为 LQR、MPC、运动学与状态方程的参考来源。

要求：

- 不主动修改。
- 作为算法和参数来源参考。
- 不作为最终运行时架构的一部分。

### 2.2 `mujoco_win`

`mujoco_win` 是当前主要仿真工程，已经实现轮腿机器人基础模型和站立控制。当前控制逻辑仍直接写在 MuJoCo 程序内部，例如通过 `wheel_leg_hooks.cc` 读取 MuJoCo 状态并写入 actuator 控制量。

当前问题：

- 控制算法与 MuJoCo 耦合。
- 控制器无法直接迁移到真机。
- ROS2 尚未接入。

阶段目标是逐步拆分为：

```text
MuJoCo
  ↓
ROS2 仿真桥接 / 硬件接口层
  ↓
Controller
```

第一阶段不要求一次性移除 `wheel_leg_hooks.cc` 中的现有控制逻辑，而是采用逐步外移策略：先保证现有仿真可运行，再逐步把状态发布、命令接收和控制器逻辑迁移到 ROS2 架构中。

### 2.3 `wetbot_ref`

`wetbot_ref` 是参考项目，可用于学习工程组织方式、控制链路拆分方式以及仿真与控制结合方式。

要求：

- 可参考设计思路。
- 不直接照搬实现。
- 保持本项目架构独立。

## 3. 长期目标架构

长期目标架构如下：

```text
遥控器
  ↓
ROS2 输入层
  ↓
状态机 / 控制模式管理
  ↓
控制器（LQR / MPC / WBC / RL / 轨迹跟踪等）
  ↓
统一机器人接口
  ↓
硬件接口层
  ├── MuJoCo
  └── STM32
```

在该架构中：

- 控制器只依赖 ROS2 接口与统一机器人状态。
- MuJoCo 只作为仿真环境或仿真硬件接口。
- STM32 只作为真实硬件接口与实时执行层。
- 遥控器、自动控制器和测试工具都应通过统一 ROS2 命令接口输入。
- 仿真模式和真机模式共享上层控制器、状态机和命令接口。

后续若进入 `ros2_control` 阶段，推荐映射关系为：

```text
仿真模式：
Controller
  ↓
MuJoCoHardware

真机模式：
Controller
  ↓
STM32Hardware
```

控制器层代码在两种模式下保持不变。

## 4. 分阶段建设方案

### 4.1 第一阶段：ROS2 与 MuJoCo 打通

第一阶段目标是先通过 Topic 桥接打通 ROS2 与 MuJoCo，不要求第一阶段直接实现完整 `ros2_control`。该阶段重点是建立数据通路、统一接口语义，并为控制器外移做准备。

目标链路：

```text
MuJoCo 模型 / mjData
  ↓
MuJoCo ROS2 桥接节点
  ↓
ROS2 Topics
  ↓
控制器节点 / 调试节点
```

主要任务：

- 在 MuJoCo 仿真侧增加 ROS2 通信能力。
- 将 MuJoCo 中的关节状态、IMU 状态和机器人状态发布到 ROS2。
- 从 ROS2 订阅控制命令，并将命令转换为 MuJoCo actuator 控制量。
- 保留现有站立控制作为过渡方案，逐步将控制目标、控制器计算和 actuator 输出从 `wheel_leg_hooks.cc` 外移。
- 明确 MuJoCo 与控制算法的边界：MuJoCo 只提供状态和执行命令，不作为控制算法的长期宿主。

第一阶段推荐接口：

| Topic | 方向 | 推荐消息类型 | 说明 |
| --- | --- | --- | --- |
| `/joint_states` | MuJoCo -> ROS2 | `sensor_msgs/msg/JointState` | 发布髋、膝、轮等关节位置与速度 |
| `/imu` | MuJoCo -> ROS2 | `sensor_msgs/msg/Imu` | 发布机体姿态、角速度和线加速度 |
| `/robot_state` | MuJoCo -> ROS2 | 后续按需定义 | 发布腿长、腿角、底盘速度、控制状态等标准消息不足以表达的状态 |
| `/joint_command` | ROS2 -> MuJoCo | 标准或轻量命令消息，后续按需定义 | 下发关节或电机力矩命令 |
| `/cmd_vel` | ROS2 -> 控制器 | `geometry_msgs/msg/Twist` | 统一速度控制输入 |
| `/control_mode` | ROS2 -> 控制器 | `std_msgs/msg/String`，后续可替换为枚举型自定义消息 | 切换站立、失能、速度控制等模式 |

第一阶段推荐实现边界：

- MuJoCo 桥接层可以读取 `mjData`、`mjModel` 和传感器数据。
- MuJoCo 桥接层可以写入 `d->ctrl` 或通过现有 actuator helper 写入控制量。
- 控制器节点不应包含 MuJoCo 类型。
- 控制器节点不应直接调用 MuJoCo API。
- 现有 `wheel_leg_hooks.cc` 可作为过渡入口，但新增逻辑应优先向“状态桥接、命令桥接、控制器节点”三个方向拆分。

第一阶段验收标准：

- 启动 MuJoCo 仿真后，ROS2 中可以看到 `/joint_states` 与 `/imu` 数据持续发布。
- 可以通过 ROS2 topic 下发简单命令，并观察 MuJoCo actuator 或机器人状态发生响应。
- 控制器相关代码与 MuJoCo API 的耦合开始减少，新增控制逻辑不再直接写入 MuJoCo UI 或渲染代码。
- 不破坏现有 MuJoCo 仿真启动与基础站立能力。

### 4.2 第二阶段：遥控器接入

第二阶段目标是将航模遥控器输入接入 ROS2。当前硬件方案已经确认：接收机为 `FlySky FS-iA6B`，输出协议为 `iBUS`，接收机直接连接 Raspberry Pi，使仿真时无需给 STM32 上电，同时保证同一个遥控器输入链路可服务仿真和真机。

当前树莓派串口已确认使用 `PASS /dev/ttyAMA3`，对应 `GPIO8/9`。本阶段核心不是继续比较多种接收机协议，而是在 Raspberry Pi 的 ROS2 节点内完成 `iBUS` 解包、通道解析、预处理和统一控制命令映射。

目标链路：

```text
FS-iA6B iBUS
  ↓
Raspberry Pi /dev/ttyAMA3
  ↓
ROS2 rc_ibus_node
  ├── /rc/channels_raw
  ├── /rc/status
  ├── /cmd_vel
  ├── /control_mode
  └── /body_cmd
  ↓
统一控制命令
  ↓
状态机 / 控制器
```

主要任务：

- 先完成树莓派串口接收联通性测试，确认 `FS-iA6B -> /dev/ttyAMA3` 确实有数据进入。
- 实现树莓派侧 `rc_ibus_node`。
- 通过 `/dev/ttyAMA3` 以 `115200 8N1` 接收 `iBUS` 字节流。
- 解析 `iBUS` 数据帧，得到 `CH1`、`CH2`、`CH3`、`CH4`、`CH5`、`CH6` 等通道原始值。
- 输出 `/rc/channels_raw` 和 `/rc/status`，支持无地面站条件下直接在树莓派上完成通道确认和排障。
- 通道映射确认需要用户现场配合，手动拨动摇杆和开关，逐一记录物理输入与通道编号的对应关系。
- 将摇杆、开关、模式选择等输入转换为统一 ROS2 命令。
- 对遥控输入执行归一化、死区处理、方向反转和失联保护。
- 输出 `/cmd_vel`、`/body_cmd`、`/control_mode` 等控制接口。
- 控制器只订阅统一命令，不关心命令来自遥控器、键盘、脚本还是上层规划器。

第二阶段验收标准：

- 在不解析控制语义之前，能够先独立确认树莓派串口已收到接收机数据。
- 遥控器输入可以在 ROS2 中被稳定读取。
- `rc_ibus_node` 能稳定识别合法 `iBUS` 帧，并在校验失败时丢弃坏帧。
- `CH1`~`CH6` 等原始通道值可以直接在树莓派 ROS2 环境中查看。
- 通过用户现场手动拨动摇杆和开关，可以确认物理输入与通道编号的映射关系。
- 遥控器节点能够发布统一控制命令。
- MuJoCo 仿真模式下可以通过遥控器改变控制目标或控制模式。
- 控制器不依赖具体遥控器协议。
- 失联、串口超时或持续坏帧时，节点会清零运动类命令并退回安全模式。

### 4.3 第三阶段：STM32 硬件通信层

第三阶段目标是建立 Raspberry Pi 与 STM32 之间的硬件通信层，使真实机器人可以接入与仿真相同的上层 ROS2 控制架构。

目标链路：

```text
ROS2
  ↓
STM32 硬件接口 / 通信节点
  ↓
串口 / CAN
  ↓
STM32
  ↓
电机驱动与传感器
```

STM32 负责：

- 电机驱动。
- 编码器读取。
- IMU 读取。
- 电流采样。
- 底层安全保护。
- 必要的实时限幅与故障处理。

Raspberry Pi 负责：

- 状态估计。
- 控制算法。
- 模式管理。
- ROS2 通信。
- 与仿真一致的控制接口维护。

主要任务：

- 定义 Raspberry Pi 与 STM32 的通信帧。
- 建立真实硬件状态到 ROS2 标准状态接口的映射。
- 建立 ROS2 控制命令到 STM32 执行命令的映射。
- 明确单位、方向、零位、限幅和安全状态。
- 在该阶段再评估是否引入 `ros2_control`，并将 STM32 封装为 `STM32Hardware`。

第三阶段验收标准：

- Raspberry Pi 可以稳定接收 STM32 上传的关节、IMU、电机和安全状态。
- ROS2 控制命令可以通过通信层发送到 STM32。
- 控制算法层不直接处理串口或 CAN 细节。
- 真机安全保护可以在通信异常或控制命令异常时进入安全状态。

### 4.4 第四阶段：仿真与实物切换

第四阶段目标是实现仿真模式与真机模式的统一切换。上层控制器、状态机和遥控器输入保持不变，只替换底层硬件接口。

目标结构：

```text
Controller
  ↓
统一机器人接口
  ↓
硬件接口层
  ├── MuJoCoHardware / MuJoCo 桥接
  └── STM32Hardware / STM32 通信层
```

主要任务：

- 对齐 MuJoCo 与 STM32 输出的状态语义。
- 对齐 MuJoCo 与 STM32 接收的命令语义。
- 统一关节命名、坐标方向、单位和时间戳。
- 建立仿真和真机的启动配置。
- 保证控制器不因底层切换而修改代码。

第四阶段验收标准：

- 同一个控制器可以在 MuJoCo 模式和真机模式下运行。
- 切换模式只需要修改启动配置或硬件接口选择。
- 仿真日志与真机日志可以按同一状态/命令语义对比。
- 控制器代码中不存在 MuJoCo API 或 STM32 通信协议依赖。

## 5. ROS2 标准接口建议

第一版接口设计遵循“标准消息优先，不足时再扩展”的原则。只有当标准消息无法表达轮腿机器人专有状态时，才定义自定义消息。

推荐 topic 如下：

| Topic | 推荐消息类型 | 说明 |
| --- | --- | --- |
| `/joint_states` | `sensor_msgs/msg/JointState` | 关节位置、速度和力矩状态 |
| `/imu` | `sensor_msgs/msg/Imu` | 机体姿态、角速度和线加速度 |
| `/rc/channels_raw` | 后续按需定义 | 遥控原始通道值、时间戳和帧有效标志 |
| `/rc/status` | 后续按需定义 | 遥控串口在线状态、错误计数和 failsafe 状态 |
| `/cmd_vel` | `geometry_msgs/msg/Twist` | 期望线速度与角速度 |
| `/control_mode` | `std_msgs/msg/String` | 控制模式，后续可替换为枚举型自定义消息 |
| `/joint_command` | 后续按需确定 | 关节或电机控制命令 |
| `/body_cmd` | 后续按需确定 | 机身附加控制目标，当前优先承载高度与转向辅助控制量 |
| `/robot_state` | 后续按需确定 | 腿长、腿角、估计状态、保护状态等综合状态 |

统一单位约定：

| 物理量 | 单位 |
| --- | --- |
| 角度 | rad |
| 角速度 | rad/s |
| 长度 | m |
| 线速度 | m/s |
| 线加速度 | m/s^2 |
| 力 | N |
| 力矩 | Nm |

命名与方向要求：

- 仿真和真机必须使用一致的关节命名。
- 左右腿、髋、膝、轮的命名应在接口文档和模型中保持一致。
- 关节正方向、轮速正方向、机体坐标系方向必须统一记录。
- 所有状态消息应带有时间戳，便于日志对齐和状态估计。

## 6. 控制算法解耦原则

控制算法模块只依赖统一机器人状态和统一控制命令，不依赖具体硬件或仿真环境。

控制器输入：

- Joint State。
- IMU。
- Robot State。
- `/cmd_vel`。
- `/body_cmd`。
- `/control_mode`。

控制器输出：

- Joint Command。
- Wheel Torque。
- Leg Command。
- 后续扩展的控制目标或力矩命令。

禁止行为：

- 控制器直接调用 MuJoCo API。
- 控制器直接读取 `mjData` 或 `mjModel`。
- 控制器直接访问 STM32 串口或 CAN。
- 控制器直接依赖具体遥控器协议。

推荐演进方式：

1. 保持现有 MuJoCo 站立控制可运行。
2. 先将 MuJoCo 状态发布到 ROS2。
3. 再从 ROS2 接收控制目标或力矩命令。
4. 将 LQR、VMC、PID、状态机等逻辑逐步拆成 ROS2 控制器节点或后续 `ros2_control` 控制器。
5. 最终让 MuJoCo 和 STM32 都只承担硬件接口职责。

## 7. 仿真与真机切换原则

仿真与真机切换的关键不是复制两套控制器，而是统一接口语义。

必须保持一致的内容：

- 关节命名。
- 坐标系定义。
- 单位约定。
- 控制命令语义。
- 状态消息语义。
- 控制模式定义。
- 安全状态定义。

允许不同的内容：

- 底层硬件接口实现。
- MuJoCo actuator 映射方式。
- STM32 通信协议。
- 真实电机驱动参数。
- 传感器滤波和标定细节。

后续进入 `ros2_control` 后，建议将差异收敛到硬件接口层：

- `MuJoCoHardware` 负责仿真状态读取和 actuator 写入。
- `STM32Hardware` 负责真实通信、状态上报和命令下发。
- Controller Manager 上层配置尽量保持一致。

## 8. 阶段验收标准汇总

第一阶段验收：

- ROS2 可以订阅到 MuJoCo 发布的 `/joint_states`。
- ROS2 可以订阅到 MuJoCo 发布的 `/imu`。
- ROS2 可以向 MuJoCo 下发控制命令。
- MuJoCo 内部控制逻辑开始向 ROS2 控制架构外移。
- 不破坏现有仿真运行能力。

第二阶段验收：

- 遥控器输入节点可用。
- 遥控器协议不泄漏到控制器。
- `/rc/channels_raw` 和 `/rc/status` 可在树莓派上直接用于调试、标定和排障。
- `/cmd_vel`、`/control_mode`、`/body_cmd` 等统一控制命令可由遥控器发布。
- 仿真模式下可通过遥控器改变机器人控制目标或控制模式。

第三阶段验收：

- Raspberry Pi 与 STM32 通信稳定。
- STM32 状态可以映射到 ROS2 标准状态接口。
- ROS2 控制命令可以下发到 STM32。
- 通信异常和命令异常有明确安全处理。

第四阶段验收：

- 同一套控制器可运行在仿真模式和真机模式。
- 切换仿真/真机不修改控制器代码。
- 仿真和真机日志可以按统一接口对比。
- 硬件差异被限制在硬件接口层。

## 9. 约束与非目标

约束：

- `matlab_sim` 作为算法参考，不主动修改。
- `wetbot_ref` 作为参考项目，不直接照搬。
- `mujoco_win` 当前可运行能力需要尽量保留。
- 第一阶段优先完成 Topic 桥接，不强制直接接入完整 `ros2_control`。
- 第一版接口标准消息优先，自定义消息按需求逐步增加。
- 第二阶段遥控器协议固定为 `FlySky FS-iA6B + iBUS`，树莓派侧接口固定为 `/dev/ttyAMA3`。

非目标：

- 本文档不定义完整自定义消息字段。
- 本文档不定义 STM32 具体通信帧格式。
- 本文档不要求立即重构全部 MuJoCo 控制代码。
- 本文档不要求立即实现真机部署。
- 本文档不要求统一所有控制算法实现细节。
