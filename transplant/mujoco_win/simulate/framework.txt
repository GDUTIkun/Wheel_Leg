整体关系

程序启动后大致是这条链路：

main.cc (line 1) 启动程序
main.cc 创建 Simulate (line 37) 对象
main.cc 开一个物理线程跑 PhysicsThread/PhysicsLoop
主线程进入 sim->RenderLoop()，这部分实现在 simulate.cc (line 2899)
物理线程里每次 mj_step 前后，会调用你加进去的 wheel_leg_hooks.cc (line 1)
所以你可以理解成：

main.cc 负责“把系统拉起来”
simulate.cc/.h 负责“官方 simulate 的大部分逻辑”
wheel_leg_hooks.cc/.h 负责“你自己的传感器/控制器逻辑”
各文件作用

main.cc (line 1)
这是程序入口。
它主要做几件事：

检查 MuJoCo 版本
扫描插件
创建 mjvCamera / mjvOption / mjvPerturb
创建 Simulate 对象
解析命令行模型路径，没传的话走默认 wheel_leg 模型
启动物理线程 PhysicsThread
主线程进入 RenderLoop
你后面如果要改“默认打开哪个模型”“程序启动行为”，一般先看这个文件。

simulate.h (line 1)
这是 Simulate 类的定义，相当于官方 UI 的“总控制器声明”。

里面放的是：

Simulate 类有哪些成员
UI 状态
相机、渲染、控制面板、历史缓冲区
RenderLoop()、Render()、Sync() 这些核心函数声明
你可以把它理解成“官方 simulate 的总数据结构说明书”。

simulate.cc (line 1)
这是最大也最核心的文件，基本上官方 simulate 的绝大多数功能都在这里。

它负责：

左右 UI 面板
profiler / sensor figure / info 面板
鼠标键盘交互
相机控制
场景渲染
截图
拖拽加载模型
UI 和 mjModel/mjData 的同步
重点函数你可以先记这几个：

RenderLoop()：主渲染循环
Render()：每帧画 UI 和场景
Sync()：把 UI 修改同步到 MuJoCo 数据
各种 ShowXXX()：画不同面板
这个文件你以后尽量少改。除非你真要扩展官方 UI 本身，不然大多数业务逻辑别塞这里。

wheel_leg_hooks.h (line 1)
这是你自己留出来的扩展接口声明文件。

它定义了 3 个入口：

OnModelLoaded
BeforeStep
AfterStep
意思很直接：

模型加载后做初始化
每次 mj_step 前做控制
每次 mj_step 后读状态/传感器
wheel_leg_hooks.cc (line 1)
这是你后面最应该常改的文件。

现在它主要做的是：

模型加载后打印一些基本信息
AfterStep 里按周期打印 hip/connect/wheel 传感器
BeforeStep 里预留控制器位置
以后你要写：

传感器读取
状态估计
PD / LQR / MPC / 轮腿控制器
记录日志
基本都往这里加。

platform_ui_adapter.h (line 1)
这是“平台 UI 适配层”的抽象接口。
它把 simulate 对窗口系统的依赖隔开，让上层只关心“我要一个窗口/UI 接口”，不关心底层是怎么开的。

platform_ui_adapter.cc (line 1)
对应上面接口的一部分通用实现，服务于 simulate.cc。

glfw_adapter.h (line 1)
这是基于 GLFW 的具体窗口适配器声明。
因为官方 simulate 用 GLFW 开窗口，所以这个文件是“GLFW 版本的 UI 后端”。

glfw_adapter.cc (line 1)
GLFW 适配器实现。
负责：

创建窗口
处理 OpenGL context
获取窗口尺寸
鼠标键盘事件接入
简单说，simulate.cc 想“画东西”和“接收输入”，很多底层动作最后都落到这里。

glfw_dispatch.h (line 1)
glfw_dispatch.cc (line 1)
这两个文件主要是 GLFW 事件分发的桥接层。
因为 GLFW 的回调常常是 C 风格函数，它们负责把这些输入事件转发回 Simulate 或 adapter 对象。

你可以把它理解成“把 GLFW 的鼠标键盘消息接到 simulate 里”。

array_safety.h (line 1)
一些小工具，主要是更安全地处理定长 char 数组、拷贝字符串之类。
这是官方 sample 常见的小辅助头文件，不是业务核心。

lodepng.h (line 1)
原本官方用它存 PNG 截图。你这版里截图逻辑已经改成了 PPM，所以这个文件现在基本可以看作遗留依赖，暂时不重要。

CMakeLists.txt (line 1)
编译入口。
负责把这些 .cc 文件编成 wheel_leg_simulate 可执行文件，并链接：

MuJoCo
GLFW
Threads
调用关系

最关键的调用链，你记这一条就够了：

main.cc (line 501) 的 main()
创建 Simulate (line 37)
启动 PhysicsThread
主线程进 RenderLoop() (line 2899)
物理线程进 PhysicsLoop
PhysicsLoop 里执行：
wheel_leg::BeforeStep(...)
mj_step(m, d)
wheel_leg::AfterStep(...)
模型加载时还会有另一条：

main.cc 里 LoadModel(...)
mj_loadXML(...)
mj_makeData(...)
mj_forward(...)
wheel_leg::OnModelLoaded(...)
所以你后面要扩展时，通常是这样分工：

改启动、默认模型、线程入口：看 main.cc
改官方 UI、面板、显示：看 simulate.cc/.h
改控制器、读传感器、打印调试：改 wheel_leg_hooks.cc/.h
改构建：看 CMakeLists.txt
你后面最实用的工作方式

如果你的目标是“保留官方 UI，同时加自己的控制和传感器逻辑”，建议遵循这个边界：

simulate.cc/.h 尽量不动
绝大多数自定义逻辑都写进 wheel_leg_hooks.cc (line 1)
也就是：

想读传感器：AfterStep
想写 d->ctrl：BeforeStep
想初始化关节/actuator/sensor id：OnModelLoaded
这样后面代码会非常稳，不容易把官方 UI 搞坏。
